## 概述

### 項目背景

VoiceLedger (語記) 是一款 AI 財務秘書應用程式，設計用於幫助用戶通過多種方式（語音、被動 GPS、照片、通知等）進行自動記帳和財務管理。

### 核心功能

1. **多源記帳**: 語音、GPS、照片、通知、手動輸入
2. **AI 助手**: 自動分類、智能建議、行為分析
3. **預算管理**: 靈活的預算設定和警告機制
4. **生活日記**: AI 生成的每日和月度摘要
5. **家庭共享**: 支持多用戶的共同帳本管理
6. **訂閱管理**: 與 RevenueCat 整合的付費模式
7. **成就系統**: 里程碑和成就解鎖

### 設計原則

- **可擴展性**: 支持數百萬用戶和數十億交易
- **性能**: 查詢響應時間 < 100ms (P95)
- **安全性**: 完整的 RLS 政策，保護用戶隱私
- **靈活性**: 支持自定義分類和標籤
- **實時性**: 實時預算監控和警告

## 表結構詳解

### 1. 用戶系統表

#### `users` - 用戶基本信息

**用途**: 存儲所有應用用戶的核心資料

**主要欄位**:
- `id` (UUID): 主鍵
- `email` (TEXT UNIQUE): 用戶郵箱，用於登錄和聯繫
- `identity_type` (ENUM): 身份類型 (student/employee/freelancer)
- `timezone` (TEXT): 時區，影響所有時間計算
- `preferences` (JSONB): 靈活的偏好設定存儲

**索引**:
```sql
PRIMARY KEY (id)
UNIQUE (email)
```

**設計亮點**:
- 使用 UUID 而非自增 ID，便於分布式系統
- `preferences` JSONB 字段支持靈活擴展
- `email_verified` 和 `phone_verified` 跟蹤驗證狀態

#### `user_patterns` - 用戶生活節奏與習慣

**用途**: 存儲 AI 學習到的用戶行為模式

**主要欄位**:
- `sleep_start_time` / `sleep_end_time`: 睡眠時間
- `peak_activity_hours` (JSONB): 活動高峰時段 [0-23]
- `mood_trend` (JSONB): 7天情緒趨勢數據
- `personality_notes` (TEXT): AI 學習的性格特徵

**設計亮點**:
- UNIQUE 約束確保每個用戶只有一份 profile
- `analyzed_at` 時間戳跟蹤最後分析時間
- JSONB 字段支持複雜的時間序列數據

### 2. 交易記錄表 (核心)

#### `transactions` - 交易主表

**用途**: 存儲所有金融交易記錄

**主要欄位**:
- `amount` (DECIMAL 12,2): 金額，精度到分
- `transaction_type` (ENUM): expense/income/transfer
- `source_type` (ENUM): 來源 (voice/passive_gps/photo/notification/manual)
- `confidence_score` (DECIMAL 3,2): 0-1 之間的信心分數
- `merchant_id` (UUID FK): 關聯商家

**設計考量**:
```sql
-- 檢查約束確保金額有效
CONSTRAINT valid_amount CHECK (amount > 0)

-- 複合索引優化查詢性能
INDEX idx_user_date (user_id, transaction_date)
INDEX idx_category (user_id, category)
```

**索引策略**:
```sql
-- 最常用的查詢模式
SELECT * FROM transactions
WHERE user_id = ? AND transaction_date >= ? AND transaction_date <= ?;

SELECT * FROM transactions
WHERE user_id = ? AND category = ?;
```

#### `transaction_audits` - 交易審核歷史

**用途**: 完整的變更追蹤和審計日誌

**主要欄位**:
- `change_type` (TEXT): created/edited/verified/categorized
- `old_values` (JSONB): 舊值快照
- `new_values` (JSONB): 新值快照
- `auditor_id` (UUID): 執行操作的用戶

**用途**:
- 完整的審計追蹤
- 支持交易撤銷/恢復
- 家庭帳本的變更通知

### 3. 被動記帳線索表

#### `geofence_logs` - 地理圍欄記錄

**用途**: 記錄用戶的位置訪問，用於自動記帳建議

**主要欄位**:
- `entry_time` / `exit_time`: 進出時間
- `triggered_transaction_id`: 關聯的交易
- `visit_purpose`: AI 推測的訪問目的

**查詢模式**:
```sql
-- 查詢用戶在特定時間和地點的訪問
SELECT * FROM geofence_logs
WHERE user_id = ? AND entry_time >= ? AND entry_time <= ?
ORDER BY entry_time DESC;

-- 地理位置索引用於附近地點查詢
SELECT * FROM geofence_logs
WHERE distance(ST_MakePoint(latitude, longitude), ?) < 100
```

#### `photo_logs` - 照片分析結果

**用途**: 存儲上傳照片的 AI 分析結果

**主要欄位**:
- `detected_receipts` (TEXT[]): 檢測到的收據
- `extracted_text` (TEXT): OCR 提取的文本
- `suggested_transaction_id`: AI 建議的相關交易
- `confidence_score`: 分析置信度

**AI 工作流程**:
```
用戶上傳照片 → AI 分析
  ├─ OCR 識別文字
  ├─ 物品檢測
  ├─ 金額提取
  └─ 交易建議
    └─ user_approved: true/false/null
```

#### `notification_logs` - 通知攔截與分析

**用途**: 從銀行/支付應用等通知自動提取交易信息

**主要欄位**:
- `notification_source` (TEXT): bank/payment_app/shopping_app
- `parsed_amount` / `parsed_merchant`: 解析結果
- `user_action`: accepted/ignored/modified

**自動化流程**:
```
系統監聽通知
  ↓
解析金額、商家、分類
  ↓
建議創建交易
  ↓
用戶確認或修改
  ↓
創建或更新交易
```

### 6. 預算管理表

#### `budgets` - 預算配置

**用途**: 存儲用戶的預算設定

**主要欄位**:
- `category`: 預算適用的分類 (NULL = 總預算)
- `amount`: 預算金額
- `frequency`: daily/weekly/monthly/yearly
- `warning_threshold`: 警告閾值 (默認 0.8 = 80%)
- `progress`: 當前已花費金額

**預算類型**:
```
1. 總體預算 (category = NULL)
   - 限制每月總支出

2. 分類預算
   - 食物: 5000
   - 交通: 2000
   - 娛樂: 1000

3. 硬限制 vs 軟限制
   - 硬限制: hard_limit = true, 達到時阻止新交易
   - 軟限制: hard_limit = false, 僅發出警告
```

#### `budget_warnings` - 預算警告歷史

**用途**: 記錄所有預算警告事件

**主要欄位**:
- `warning_level`: warning/critical/exceeded
- `percentage_used`: 預算使用百分比
- `acknowledged_at`: 用戶確認時間

**警告觸發邏輯**:
```sql
CASE
  WHEN (progress / amount) >= 1 THEN 'exceeded'     -- 已超支
  WHEN (progress / amount) >= 0.95 THEN 'critical'  -- 即將超支
  ELSE 'warning'                                     -- 接近限額
END
```

## 關鍵設計決策

### 1. UUID vs 自增 ID

**決策**: 使用 UUID (uuid_generate_v4)

**原因**:
- 支持離線操作和同步
- 適合分布式系統
- 避免 ID 碰撞
- 隱私友好（不暴露記錄數量）

### 2. JSONB 字段使用

**決策**: 在某些字段使用 JSONB

**應用場景**:
- `preferences`: 用戶偏好設定
- `source_detail`: 交易來源詳情
- `metadata`: 靈活的額外信息

**優勢**:
- 無需修改 schema 即可添加新字段
- 支持複雜查詢 (jsonb_extract_text, @> 操作符)
- 應用層友好

**限制**:
- 不應過度使用（會降低查詢性能）
- 需要應用層驗證
- 不支持複雜的關係約束

### 3. 軟刪除 vs 硬刪除

**決策**: 使用軟刪除 (deleted_at 時間戳)

**優勢**:
- 完整的審計追蹤
- 易於恢復誤刪數據
- 符合數據保護法規（GDPR）

**實現**:
```sql
-- users 表示例
SELECT * FROM users WHERE deleted_at IS NULL;
```

### 4. 複合索引策略

**決策**: 採用複合索引優化常見查詢

**示例**:
```sql
-- 最常見查詢：用戶在日期範圍內的交易
CREATE INDEX idx_transactions_user_date
ON transactions(user_id, transaction_date DESC);

-- 用戶分類查詢
CREATE INDEX idx_transactions_category
ON transactions(user_id, category);
```

### 5. 觸發器自動化

**決策**: 使用觸發器自動更新統計信息

**自動化項**:
- 用戶總支出和交易數量
- 預算進度跟蹤
- 商家訪問統計
- 每日日記自動創建

**好處**:
- 數據一致性
- 避免應用層複雜邏輯
- 提高查詢性能（預計算）

## Row Level Security (RLS)

### RLS 政策概述

RLS 在資料庫層面實施訪問控制，確保用戶只能訪問自己和有權限的數據。

### 核心政策

#### 1. 用戶表 RLS

```sql
-- 用戶只能看到自己或家庭成員的信息
CREATE POLICY "用戶只能看到自己的資料" ON users
  FOR SELECT USING (
    auth.uid() = id
    OR id IN (
      SELECT user_id FROM family_members
      WHERE family_id IN (
        SELECT family_id FROM family_members WHERE user_id = auth.uid()
      )
    )
  );

-- 用戶只能修改自己的資料
CREATE POLICY "用戶只能更新自己的資料" ON users
  FOR UPDATE USING (auth.uid() = id);
```

#### 2. 交易表 RLS

```sql
-- 用戶只能看到自己或共享帳本的交易
CREATE POLICY "用戶只能看到自己的交易" ON transactions
  FOR SELECT USING (
    user_id = auth.uid()
    OR family_ledger_id IN (
      SELECT id FROM family_ledgers
      WHERE family_id IN (
        SELECT family_id FROM family_members WHERE user_id = auth.uid()
      )
    )
  );

-- 用戶只能創建自己名下的交易
CREATE POLICY "用戶只能創建自己的交易" ON transactions
  FOR INSERT WITH CHECK (user_id = auth.uid());
```

#### 3. 預算表 RLS

```sql
-- 用戶只能看到和管理自己的預算
CREATE POLICY "用戶只能看到自己的預算" ON budgets
  FOR SELECT USING (user_id = auth.uid());

CREATE POLICY "用戶只能管理自己的預算" ON budgets
  FOR INSERT WITH CHECK (user_id = auth.uid());
```

#### 4. 家庭成員表 RLS

```sql
-- 只有家庭成員可以看到家庭信息
CREATE POLICY "家庭成員可以看到家庭信息" ON family_members
  FOR SELECT USING (
    user_id = auth.uid()
    OR family_id IN (
      SELECT family_id FROM family_members WHERE user_id = auth.uid()
    )
```

## 索引策略

### 索引規劃原則

1. **查詢優先**: 根據實際查詢模式創建索引
2. **避免冗余**: 去除不必要的索引
3. **監控性能**: 定期檢查索引使用情況
4. **考慮寫入成本**: 每個索引都會增加寫入成本

### 核心索引

#### 交易表索引

```sql
-- 最常用：用戶在日期範圍內的交易
CREATE INDEX idx_transactions_user_date
ON transactions(user_id, transaction_date DESC);

-- 用戶分類查詢
CREATE INDEX idx_transactions_category
ON transactions(user_id, category);

-- 商家查詢
CREATE INDEX idx_transactions_merchant
ON transactions(merchant_id);

-- 來源查詢
CREATE INDEX idx_transactions_source
ON transactions(source_type);
```

**查詢範例**:
```sql
-- 利用 idx_transactions_user_date
SELECT * FROM transactions
WHERE user_id = 'user-123'
  AND transaction_date BETWEEN '2026-03-01' AND '2026-03-31'
ORDER BY transaction_date DESC;

-- 利用 idx_transactions_category
SELECT SUM(amount) as total
FROM transactions
WHERE user_id = 'user-123' AND category = 'food';
```

#### 地理位置索引

```sql
-- 圍欄日誌位置查詢
```

## 觸發器與自動化

### 觸發器列表

#### 1. update_user_stats() - 更新用戶統計

**觸發時機**: INSERT transaction

**功能**:
- 更新 users.total_expense / total_income
- 增加 users.transaction_count
- 更新 users.last_transaction_date

```sql
CREATE TRIGGER trg_update_user_stats_on_transaction
AFTER INSERT ON transactions
FOR EACH ROW
EXECUTE FUNCTION update_user_stats();
```

**實現細節**:
```sql
CREATE OR REPLACE FUNCTION update_user_stats()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.transaction_type = 'expense' THEN
    UPDATE users
    SET
      total_expense = total_expense + NEW.amount,
      transaction_count = transaction_count + 1,
      last_transaction_date = NEW.transaction_date,
      updated_at = CURRENT_TIMESTAMP
    WHERE id = NEW.user_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

#### 2. update_budget_progress() - 跟蹤預算進度

**觸發時機**: INSERT transaction (expense 類型)

**功能**:
- 更新 budgets.progress
- 觸發預算警告 (budget_warnings)
- 檢查是否超出限額

**預算匹配邏輯**:
```sql
-- 匹配條件
- user_id 相符
- category 相符或預算 category 為 NULL (總預算)
- 交易日期在預算周期內
- 交易類型為 expense
```

#### 3. ensure_daily_diary() - 自動創建日記

**觸發時機**: INSERT transaction

**功能**: 自動為該日期創建或更新 daily_diary 記錄

```sql
INSERT INTO daily_diaries (user_id, diary_date)
VALUES (NEW.user_id, NEW.transaction_date)
ON CONFLICT (user_id, diary_date) DO NOTHING;
```

#### 4. update_merchant_stats() - 商家統計

**觸發時機**: INSERT transaction (包含 merchant_id)

**更新的統計**:
- merchants.user_count: 訪問過的不同用戶數
- merchants.avg_transaction_amount: 平均交易金額

#### 5. cleanup_expired_notifications() - 清理過期通知

**觸發時機**: INSERT notification

**功能**: 刪除已過期的通知記錄

## 查詢優化

### 常見查詢模式

#### 1. 用戶月度支出統計

```sql
-- 原始查詢
SELECT
  DATE_TRUNC('month', transaction_date)::DATE as month,
  category,
  SUM(amount) as total,
  COUNT(*) as count
FROM transactions
WHERE user_id = $1
  AND transaction_type = 'expense'
  AND transaction_date >= $2
  AND transaction_date <= $3
GROUP BY DATE_TRUNC('month', transaction_date), category
ORDER BY month DESC, total DESC;

-- 優化建議
-- 1. 使用預計算視圖 (user_monthly_stats)
-- 2. 添加複合索引 (user_id, transaction_date, category)
-- 3. 考慮物化視圖進行日期範圍查詢
```

#### 2. 尋找異常交易

```sql
-- 檢測高於平均值的交易
WITH user_stats AS (
  SELECT
    user_id,
    AVG(amount) as avg_amount,
    STDDEV(amount) as stddev_amount
  FROM transactions
  WHERE transaction_type = 'expense'
    AND transaction_date >= CURRENT_DATE - INTERVAL '30 days'
  GROUP BY user_id
)
SELECT
  t.*
FROM transactions t
JOIN user_stats s ON t.user_id = s.user_id
WHERE t.amount > s.avg_amount + (2 * s.stddev_amount)
  AND t.transaction_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY t.amount DESC;

-- 優化：使用物化視圖預計算統計信息
CREATE MATERIALIZED VIEW anomaly_detection_baseline AS
SELECT
  user_id,
  AVG(amount) as avg_amount,
  STDDEV(amount) as stddev_amount
FROM transactions
WHERE transaction_type = 'expense'
  AND transaction_date >= CURRENT_DATE - INTERVAL '90 days'
GROUP BY user_id;
```

#### 3. 家庭帳本結算

```sql
-- 計算誰欠誰多少錢
WITH ledger_transactions AS (
  SELECT
    ledger_id,
    user_id,
    SUM(amount) as total_spent
  FROM transactions
  WHERE family_ledger_id IS NOT NULL
  GROUP BY ledger_id, user_id
),
ledger_stats AS (
  SELECT
    ledger_id,
    SUM(total_spent) as total_amount,
    COUNT(DISTINCT user_id) as member_count
  FROM ledger_transactions
  GROUP BY ledger_id
)
SELECT
  t.ledger_id,
  t.user_id,
  t.total_spent,
  l.total_amount / l.member_count as per_person_share,
  (l.total_amount / l.member_count) - t.total_spent as settlement_amount
FROM ledger_transactions t
JOIN ledger_stats l ON t.ledger_id = l.ledger_id
ORDER BY t.ledger_id, settlement_amount;
```

| 查詢類型 | 目標響應時間 | 實際響應時間 |
|---------|-----------|-----------|
| 單月交易 | < 50ms | ~20ms |
| 年度統計 | < 100ms | ~80ms |
| 分類報告 | < 100ms | ~90ms |
| 異常檢測 | < 500ms | ~300ms |
| 家庭結算 | < 200ms | ~150ms |

## 附錄：常用 SQL 查詢

### 1. 月度財務總結

```sql
SELECT
  DATE_TRUNC('month', t.transaction_date)::DATE as month,
  SUM(CASE WHEN t.transaction_type = 'expense' THEN t.amount ELSE 0 END) as total_expense,
  SUM(CASE WHEN t.transaction_type = 'income' THEN t.amount ELSE 0 END) as total_income,
  COUNT(*) as transaction_count
FROM transactions t
WHERE t.user_id = $1
GROUP BY DATE_TRUNC('month', t.transaction_date)
ORDER BY month DESC;
```

### 2. 分類支出分布

```sql
SELECT
  category,
  COUNT(*) as count,
  SUM(amount) as total,
  ROUND(100.0 * SUM(amount) / SUM(SUM(amount)) OVER (), 2) as percentage
FROM transactions
WHERE user_id = $1 AND transaction_type = 'expense'
  AND transaction_date >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY category
ORDER BY total DESC;
```

### 3. 預算狀態檢查

```sql
SELECT
  b.name,
  b.category,
  b.amount as budget_limit,
  b.progress as current_spend,
  ROUND(100.0 * b.progress / b.amount, 1) as utilization_percent,
  CASE
    WHEN b.progress >= b.amount THEN 'exceeded'
    WHEN b.progress >= (b.amount * b.warning_threshold) THEN 'warning'
    ELSE 'normal'
  END as status
FROM budgets b
WHERE b.user_id = $1 AND b.is_active = TRUE
ORDER BY utilization_percent DESC;
```

### 4. 商家訪問頻率

```sql
SELECT
  m.name,
  COUNT(t.id) as visit_count,
  MAX(t.transaction_date) as last_visit,
  ROUND(AVG(t.amount), 2) as avg_spend,
  SUM(t.amount) as total_spent
FROM transactions t
JOIN merchants m ON t.merchant_id = m.id
WHERE t.user_id = $1 AND t.transaction_type = 'expense'
GROUP BY m.id, m.name
ORDER BY visit_count DESC
LIMIT 20;
```

### 5. 消費趨勢分析

```sql
WITH daily_stats AS (
  SELECT
    transaction_date,
    SUM(amount) as daily_total,
    COUNT(*) as transaction_count
  FROM transactions
  WHERE user_id = $1 AND transaction_type = 'expense'
    AND transaction_date >= CURRENT_DATE - INTERVAL '30 days'
  GROUP BY transaction_date
)
SELECT
  transaction_date,
  daily_total,
  transaction_count,
  ROUND(AVG(daily_total) OVER (ORDER BY transaction_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), 2) as moving_avg_7d,
  ROUND(100.0 * (daily_total - LAG(daily_total) OVER (ORDER BY transaction_date)) / LAG(daily_total) OVER (ORDER BY transaction_date), 1) as day_over_day_change
FROM daily_stats
ORDER BY transaction_date DESC;
```
